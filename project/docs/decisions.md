# Architectural Decisions

1. **Registry backend (local JSON index with optional MLflow mirror).** Selected a filesystem-backed registry stored under `ARTIFACT_DIR` to simplify bootstrap and guarantee deployability in air-gapped runtimes. MLflow integration is gated by `USE_MLFLOW` so teams can opt-in without changing code paths.
2. **Model format (scikit-learn + joblib for baseline).** The initial use-case favours tabular data and quick iteration. Joblib artifacts keep dependencies light. Torch/TensorFlow backends can slot in later via registry metadata without rewriting services.
3. **Transport protocols (JSON over HTTP, Parquet for batch).** REST with JSON keeps prediction clients simple and contract-driven. Batch inference writes Parquet for efficient columnar analytics and straightforward downstream ingestion.
4. **Deterministic splits and seeding.** Global `SEED` environment variable seeds Python, NumPy, and scikit-learn to ensure reproducibility. `src/data/split.py` performs stratified splits so experiments are comparable.
5. **Strict contract testing.** Contract-focused pytest suites (`tests/contract`) fail the CI pipeline if schemas or I/O columns change unintentionally, protecting downstream integrations.
6. **Feature pipeline as explicit ColumnTransformer.** Centralising feature logic in a saved transformer guarantees training/serving parity and keeps preprocessing auditable.
7. **Error envelopes are immutable.** Online services always respond with the documented `{status, error_code, message, request_id}` payload to allow consistent retry/error handling across clients.
8. **Single source of truth for contracts and runtime constants.** Project-wide identifiers (model name, primary metric, data file names) now live in `src/common/settings.py`, while schema-bound limits reside in `src/data/schema.py`, preventing divergence across services and tests.
9. **Ledger source of truth (JournalEntry/JournalLine).** JournalEntry/JournalLine are the authoritative record; Transaction remains a compatibility layer for legacy reads/writes until Phase 3 convergence. This keeps double-entry integrity as the system of record.
10. **Money schedule account boundary (explicit mapping, no implicit reuse).** Money schedule accounts remain a separate domain; any linkage to accounting accounts must be explicit via a mapping/interface and handled in services. This prevents forecast-only accounts from silently leaking into the ledger.
11. **ML runtime asset location (runtime-only under finance_app/ml or external service).** Runtime artifacts belong under `finance_app/ml` (or an external ML service) and training artifacts live outside the deployable runtime footprint; no new joblib artifacts should land in the repo root. This keeps deployables lean and avoids accidental runtime coupling.
12. **Phase 3 schema scaffolding (compat + explicit mappings).** Added `transaction_journal_link` and `money_schedule_account_link` plus user-scoped composite indexes to implement the Phase 3 ledger convergence and explicit money schedule boundary decisions without changing runtime behavior. Reconciliation SQL lives in `scripts/reconcile_ledger_convergence.sql`; compat population is handled by services/backfills.
13. **Convergence confidence policy (exact/strong/weak).** Automatic transaction-to-journal linking is allowed only for `exact` and `strong` confidence where at least one stable provenance identifier exists and candidate cardinality is strictly 1:1. Heuristic or ambiguous matches are persisted as `weak` candidates for review and never auto-linked.
14. **Canonical reporting source policy (no mixed mode).** Ranked accounting reports (`/accounting/tb/monthly`, statement data/export/pdf, TB PDF) now reject `mixed` source mode. Canonical totals are journal-based; legacy-only rows are surfaced via coverage metrics and reconciliation output rather than merged into report totals.
15. **Capability-based schema guards for sensitive operations.** Sensitive flows (`/upload_csv`, TB reset, backfill/reconcile commands, admin mutating actions) are blocked when required table/index/constraint capabilities are missing, independent of alembic revision string.
16. **Deterministic CSV ingestion and idempotency.** CSV import defaults to journal-first and applies both file-level (`csv_import_batch`) and row-level (`csv_import_row`) dedupe keys across files. Duplicate rows are skipped deterministically and reported in import summaries.
17. **Write-mode contract and atomicity.** `LEDGER_WRITE_MODE` is restricted to `journal|dual|legacy`; dual write requires creating the journal entry, legacy row, and `transaction_journal_link` atomically in one DB transaction.
18. **Admin/TB destructive-action safety.** Destructive admin actions and TB reset require CSRF, typed/explicit confirmation, cooldown timestamp validation, and append-only audit rows (`admin_action_audit`) for post-incident traceability.
